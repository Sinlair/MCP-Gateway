# PRD: MCP Gateway

## 1. Introduction / Overview

MCP Gateway 是一个面向 AI 应用开发者与平台管理员的统一接入网关。它位于调用方与多个上游 MCP Server 之间，对外提供单一入口，对内负责上游服务注册、客户端鉴权、会话上下文管理、工具发现、请求路由与策略控制。

当前痛点是：多个 MCP Server 需要逐个对接，接入方式分散；不同服务的鉴权、工具暴露和错误格式不一致；缺少统一治理能力，难以做权限控制、上游切换和问题排查。MCP Gateway 的目标是把这些差异收敛到网关层。

## 2. Assumptions

本 PRD 基于以下一期假设：

- 目标优先级：统一接入多个 MCP Server，对外提供单一网关入口。
- 目标用户：同时覆盖 AI 应用开发者和平台管理员。
- 一期范围：既支持下游客户端接入网关，也支持网关代理到上游 MCP Server。
- 一期能力：会话管理、服务注册、请求路由、鉴权、工具发现、工具级策略控制。
- 文档深度：标准 PRD，附带 Java + DDD 的技术约束与设计原则。

## 3. Goals

- 为调用方提供一个稳定的 MCP 接入地址，避免点对点集成多个上游服务。
- 允许管理员通过配置或管理接口注册、启停、更新上游 MCP Server。
- 统一暴露工具发现和工具调用能力，屏蔽上游实现差异。
- 在网关层完成客户端鉴权和工具级访问控制，未授权请求不得进入上游服务。
- 支持基础可观测性，能够定位“谁在什么时间调用了哪个工具，路由到了哪个上游”。
- 将网关额外处理开销控制在 p95 < 100ms（不含上游工具执行时长）。

## 4. User Stories

### US-001: 注册和管理上游 MCP Server
**Description:** 作为平台管理员，我希望注册、启停和更新上游 MCP Server，这样我可以在不改代码的情况下维护网关可用能力。

**Acceptance Criteria:**
- [ ] 管理员可以新增上游服务配置，至少包含 `serverCode`、`name`、`baseUrl`、`transportType`、`authMode`、`enabled`。
- [ ] 同一环境内 `serverCode` 不允许重复。
- [ ] 禁用的上游服务不会参与工具发现和请求路由。
- [ ] 非法 URL、缺失必填项或不支持的传输类型会被明确拒绝。
- [ ] `mvn test` 通过。

### US-002: 客户端鉴权与会话上下文建立
**Description:** 作为 AI 应用开发者，我希望通过统一凭证访问网关，这样我不需要为每个上游服务分别管理认证方式。

**Acceptance Criteria:**
- [ ] 网关支持 `API Key` 或 `Bearer Token` 方式校验客户端身份。
- [ ] 鉴权成功后，网关为请求建立或恢复会话上下文，至少包含 `requestId`、`clientId`、`sessionId`。
- [ ] 未携带凭证、凭证无效或无权限的请求返回明确的 `401` 或 `403`。
- [ ] 未授权请求不会转发到任何上游服务。
- [ ] `mvn test` 通过。

### US-003: 统一工具发现
**Description:** 作为 AI 应用开发者，我希望从网关一次性获取可用工具列表，这样我可以避免逐个查询多个上游服务。

**Acceptance Criteria:**
- [ ] 网关可以聚合所有已启用上游服务的工具元数据。
- [ ] 返回结果至少包含 `toolName`、`serverCode`、`description`、`inputSchema`。
- [ ] 被禁用、鉴权失败或健康检查失败的上游服务不会出现在工具发现结果中。
- [ ] 相同工具名来自不同上游时，结果中能够明确区分来源。
- [ ] `mvn test` 通过。

### US-004: 工具调用路由
**Description:** 作为 AI 应用开发者，我希望通过网关调用指定工具，这样我只面向一个入口即可完成工具执行。

**Acceptance Criteria:**
- [ ] 网关能够根据工具标识、绑定关系或路由规则将请求转发到正确的上游服务。
- [ ] 网关会透传必要参数，并对上游响应做统一封装。
- [ ] 上游超时、4xx、5xx 和协议错误会被映射为统一的网关错误模型。
- [ ] 每次调用都产生日志，至少包含 `requestId`、`clientId`、`toolName`、`serverCode`、`latency`、`status`。
- [ ] `mvn test` 通过。

### US-005: 工具级访问控制
**Description:** 作为平台管理员，我希望按客户端或客户端组控制可调用工具，这样我可以限制高风险工具的使用范围。

**Acceptance Criteria:**
- [ ] 管理员可以为客户端配置工具白名单或黑名单策略。
- [ ] 策略命中时，网关在调用上游前做出放行或拒绝决策。
- [ ] 被拒绝的请求返回明确错误码和可审计原因。
- [ ] 策略变更后，新请求无需重启服务即可生效。
- [ ] `mvn test` 通过。

### US-006: 上游健康检查与状态查询
**Description:** 作为平台管理员，我希望查看上游服务当前健康状态，这样我可以在故障发生时快速定位异常节点。

**Acceptance Criteria:**
- [ ] 网关提供状态查询能力，至少展示 `serverCode`、`enabled`、`healthStatus`、`lastCheckedAt`。
- [ ] 健康检查失败的上游不会继续参与新请求路由。
- [ ] 管理员可以主动触发一次状态刷新。
- [ ] `mvn test` 通过。

## 5. Functional Requirements

1. **FR-1:** 系统必须提供上游 MCP Server 注册、更新、启用、禁用和查询能力。
2. **FR-2:** 系统必须至少支持网络型上游接入方式；一期默认支持 HTTP/SSE 风格连接，不要求支持 `stdio`。
3. **FR-3:** 系统必须在所有网关入口处执行客户端鉴权，并在鉴权成功后创建统一请求上下文。
4. **FR-4:** 系统必须生成全链路唯一 `requestId`，并在日志、错误响应和追踪信息中透出。
5. **FR-5:** 系统必须聚合已启用且健康的上游工具列表，对外暴露统一的工具发现接口。
6. **FR-6:** 系统必须保存工具与上游服务的映射关系，并支持基于配置的路由决策。
7. **FR-7:** 当客户端发起工具调用时，系统必须在网关层完成鉴权、策略校验、路由选择和请求转发。
8. **FR-8:** 系统必须将上游错误转换为统一响应模型，避免将内部异常栈直接暴露给调用方。
9. **FR-9:** 系统必须支持工具级访问控制，至少覆盖“按客户端配置白名单/黑名单”。
10. **FR-10:** 系统必须记录审计日志，至少包含调用方、目标工具、目标上游、调用结果和耗时。
11. **FR-11:** 系统必须提供上游健康状态查询，并在上游不可用时停止继续路由新请求。
12. **FR-12:** 系统必须支持环境隔离配置，至少区分 `dev`、`test`、`prod`。
13. **FR-13:** 系统必须允许为上游配置超时时间；一期默认不做自动重试，以避免非幂等工具的副作用放大。
14. **FR-14:** 系统必须提供管理员使用的管理接口，与对外网关调用接口分离。

## 6. Non-Goals (Out of Scope)

- 一期不提供完整的 Web 管理后台，默认通过配置文件或管理 API 操作。
- 一期不做复杂的工作流编排、多工具链式执行或 Agent 编排引擎。
- 一期不做计费、结算、套餐或配额计量。
- 一期不支持 `stdio` 型本地进程托管，只处理网络可访问的上游服务。
- 一期不做跨地域容灾、全局调度和多活部署。
- 一期不做细粒度 RBAC 后台权限模型，先聚焦客户端级与工具级访问控制。

## 7. Design Considerations

- 对外建议区分两类接口：
  - 网关接口：面向 AI 应用开发者，用于鉴权、工具发现、工具调用。
  - 管理接口：面向管理员，用于上游服务管理、策略管理和状态查询。
- 网关响应应保持稳定，避免把上游服务的字段差异直接暴露给客户端。
- 对于同名工具，建议采用 `serverCode + toolName` 的组合标识，避免路由歧义。
- 所有错误响应都应包含 `requestId`，便于调用方与运维侧联合排障。

## 8. Technical Considerations

- 技术栈基线：Java 21、Spring Boot 3、Maven 多模块工程。
- 推荐模块划分：
  - `mcp-gateway-app`：启动、配置、装配。
  - `mcp-gateway-domain`：领域模型、聚合、值对象、领域服务、仓储端口。
  - `mcp-gateway-infrastructure`：数据库、缓存、上游 MCP 适配器、外部集成。
  - `mcp-gateway-trigger`：REST/SSE/WebSocket 入口。
  - `mcp-gateway-types`：共享枚举、异常、响应模型。
- DDD 建模建议按限界上下文拆分为：`upstream-registry`、`session`、`tool-catalog`、`routing`、`access-control`。
- 领域层只表达业务规则，不直接依赖 HTTP、数据库和第三方协议细节；协议转换和外部调用放在基础设施层。
- 推荐 MySQL 持久化上游配置与策略，Redis 可选用于会话和工具元数据缓存。
- 所有敏感配置必须通过环境变量或外部配置中心注入，不得硬编码在仓库中。

## 9. Success Metrics

- 开发者只使用 1 个网关地址和 1 套凭证，即可完成对 3 个以上上游 MCP Server 的接入。
- 新增一个上游服务的平均接入时间小于 30 分钟，且无需发布业务代码。
- 100% 未授权请求在网关层被拦截，不进入上游服务。
- 工具发现结果准确率达到 95% 以上，不出现已禁用服务仍被返回的情况。
- 网关额外处理开销 p95 小于 100ms（不含上游执行时长）。

## 10. Open Questions

- 一期是否需要多租户隔离，还是先按单租户平台建设？
- 一期是否必须支持流式响应透传，还是先保证普通请求/响应链路？
- 工具路由规则是否只基于静态映射，还是需要预留按标签、成本或延迟动态选路？
- 是否需要把调用审计输出到外部日志/监控平台，还是先保存在本地日志体系中？
